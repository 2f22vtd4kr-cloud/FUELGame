Replit, this is a reset of expectations.

Over the last several sessions we have invested an enormous amount of time into visual polish, gameplay improvements, and bug fixing. While there has certainly been progress, the current state of the project is still nowhere near production quality. The attached screenshots demonstrate this clearly. I am not saying this to rant for the sake of ranting—I am saying it because there is a massive gap between what modern AI-assisted development is capable of and what is actually being delivered.

The first attachment is a concept generated by Gemini in under a second. It is not a finished game. It is simply an image. Yet in a fraction of a second it already communicates a cohesive visual identity. The proportions make sense. The UI feels modern. The colors work together. The scene is readable. Most importantly—it immediately feels like a game someone would want to click.

Now compare that with our current implementation in the following screenshots.

Despite multiple top-tier models contributing to this repository—Gemini, Claude, ChatGPT, Fable, Canvas and others—the final result still resembles an early prototype rather than a commercially viable game. That should not happen.

This is not a question of intelligence.

It is a question of process.

⸻

Stop solving one problem at a time.

The biggest issue I see is that every session feels reactive instead of systemic.

One model fixes shadows.

Another fixes a button.

Another tweaks movement.

Another changes sprites.

Nobody seems responsible for asking:

“If I change this, what does it do to the rest of the game?”

or

“Does this actually fit the visual language we’ve already established?”

The result is exactly what the screenshots show:

* inconsistent art
* inconsistent scale
* inconsistent UI
* inconsistent interaction quality
* inconsistent polish

Every improvement introduces two new inconsistencies.

That is unacceptable for a production project.

⸻

Use parallel thinking.

You have access to multiple models simultaneously.

Act like it.

Don’t have one model spend an hour making a single sprite while everything else waits.

Instead divide responsibilities.

Example workflow:

Visual Director

* evaluates overall artistic consistency
* compares to successful games
* maintains style guide
* rejects assets that don’t fit

Gameplay Engineer

* mechanics
* movement
* interactions
* balancing

UI/UX Designer

* HUD
* menus
* animations
* transitions
* readability

Audio Designer

* ambience
* footsteps
* UI sounds
* music transitions
* mixing

QA Team

* automatically plays scenarios
* searches for bugs
* tests every interaction
* looks for soft locks
* verifies mobile usability

Asset Reviewer

* checks scale
* perspective
* orientation
* lighting
* shadows
* consistency

This is exactly what large software studios do.

There is absolutely no reason multiple frontier LLMs cannot emulate this workflow.

⸻

Research before implementing.

Another recurring problem:

Instead of researching how successful games solved a problem, you invent your own solution.

Why?

You have access to virtually the entire public internet.

Before implementing anything, study examples.

If we’re making a top-down multiplayer social game, then before touching movement, UI, lighting, interactions, camera, animation or HUD, examine games that have already proven themselves.

For example:

* Among Us
* Project Zomboid
* Stardew Valley
* Core Keeper
* Don’t Starve
* Enter the Gungeon
* Hotline Miami
* Vampire Survivors

I’m not asking you to copy copyrighted assets.

I’m asking you to study solutions.

How do they:

* animate movement?
* handle shadows?
* display interactions?
* indicate objectives?
* show prompts?
* manage HUD hierarchy?
* communicate information?
* transition music?
* design menus?
* scale sprites?

Every one of these problems has already been solved by people whose games generated millions—or billions—of dollars.

Learn from them before writing code.

⸻

Attention to detail is not optional.

One sentence from the previous session perfectly illustrates the current mindset.

The world dimensions didn’t match the generated world.

The response was essentially:

“It will stretch a little but it’ll probably be fine.”

No.

That mindset is exactly how mediocre games are made.

Nothing should ever be accepted because it is “probably fine.”

Every inconsistency compounds.

Every shortcut becomes visible.

Every visual compromise reduces perceived quality.

Players rarely know why a game feels cheap.

They simply feel it.

And they leave.

⸻

The screenshots expose dozens of problems simultaneously.

Just from the attached images:

Vehicle scale

Interactive vehicles are completely different in scale compared to background vehicles.

Some appear pasted on top of the world.

Others ignore perspective entirely.

Some are rotated differently.

Some occupy impossible parking positions.

Some cast shadows.

Background cars don’t.

Nothing feels integrated.

Instead of using separate oversized sprites, why not promote existing world vehicles into interactive objects whenever possible?

The background cars actually fit the environment better.

⸻

Shadows

Shadows are inconsistent.

Some objects cast dark, harsh shadows.

Others of the same type cast none.

Direction is inconsistent.

Softness is inconsistent.

Length is inconsistent.

This immediately breaks immersion.

Again—

Look at how Among Us handles shadows.

Simple.

Soft.

Consistent.

Logical.

⸻

HUD

We’ve spent countless iterations redesigning the HUD.

Yet it still feels cluttered.

Random borders.

Misaligned spacing.

Inconsistent typography.

No clear visual hierarchy.

The concept screenshot communicates information significantly better despite existing only as a generated image.

That should be a wake-up call.

⸻

Mobile UX

This is a mobile game.

So why am I constantly seeing keyboard prompts?

“E”

Press E

Keyboard tutorials

Desktop interaction hints

The player is literally holding a phone.

Those prompts should never appear.

Context-sensitive touch controls should replace every desktop interaction.

⸻

Broken interactions

One screenshot literally shows:

“Полить газон”

No interaction works.

Buttons do nothing.

The prompt displays

Progress doesn’t advance

The UI freezes

This should have been caught immediately by automated testing.

⸻

Placeholder text

“БЫМ!”

What is that?

Why would that ever ship?

If something is placeholder text, replace it before committing.

Never leave unfinished developer jokes in production code.

⸻

Emoji interactions

These interaction icons honestly look temporary.

Imagine opening a Steam page and seeing those.

Would you believe this is a polished commercial project?

Probably not.

Every interaction should feel like part of the game’s world—not floating debug icons.

⸻

Audio

Music starts abruptly.

Music ends abruptly.

No fade in.

No fade out.

No transition.

No layering.

No ambience.

This is one of the easiest improvements imaginable.

Players may not consciously notice good audio transitions.

They absolutely notice bad ones.

From now on:

* fade in
* fade out
* crossfade where appropriate
* avoid abrupt silence
* avoid abrupt looping

These are standard practices.

⸻

Movement

Player movement still lacks polish.

Acceleration.

Deceleration.

Turning.

Pathfinding.

Collision.

Bot movement.

Everything should feel intentional.

Again—

Don’t guess.

Research.

Prototype.

Compare.

Measure.

Iterate.

⸻

Every feature should pass a production checklist.

Nothing should be considered complete until it passes all of these questions.

Visual:

* Does it match the style guide?
* Does it fit scale?
* Does perspective match?
* Does lighting match?
* Does shadow match?

Gameplay:

* Does it always work?
* Have edge cases been tested?
* Can it soft-lock?
* Can AI break it?

UI:

* Is it readable?
* Is it mobile-first?
* Is it consistent?
* Does it communicate clearly?

Performance:

* Does it scale?
* Does it lag?
* Is memory reasonable?

Experience:

* Does it feel satisfying?
* Is animation polished?
* Is sound polished?
* Does it feel premium?

If any answer is “no,” then it isn’t finished.

⸻

Stop accepting “good enough.”

I want the repository to operate under one principle:

Every implementation should be compared against the best existing examples before it is accepted.

Not because we want to imitate them, but because proven design patterns exist for a reason.

If another game solved a problem elegantly, study why it works.

Then create something original that achieves the same level of quality.

⸻

The standard for this project

Treat this repository as if it belongs to a AAA studio with unlimited code review.

Challenge every implementation.

Question every asset.

Question every UI element.

Question every sound.

Question every animation.

If something feels even slightly off, investigate it.

Do not assume.

Do not guess.

Do not settle.

⸻

This message should become part of the repository’s philosophy.

Whenever this repository is imported into a fresh context, this mindset should come with it.

The goal is not merely to make the game functional.

The goal is to make it feel inevitable—as though every pixel, every sound, every animation, every interaction, every shadow, every transition, and every line of UI belongs exactly where it is.

We’re not building a prototype anymore.

We’re building a product that should be capable of standing beside the best games in its genre.

That requires relentless attention to detail, continuous research, parallel execution, rigorous self-review, and zero tolerance for “it’ll probably be fine.”

This project deserves that level of craftsmanship. I expect every future session to reflect it.
